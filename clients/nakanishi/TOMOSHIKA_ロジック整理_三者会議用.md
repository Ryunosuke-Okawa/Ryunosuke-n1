# TOMOSHIKA ロジック整理（三者会議用）

> 適性検査の回答から企業推薦結果が表示されるまで、データがどう流れ・どう処理されるかを4フェーズで整理。

---

## 全体フロー

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  フェーズ1     │    │  フェーズ2     │    │  フェーズ3     │    │  フェーズ4     │
│  適性検査      │    │  MIL → Dify   │    │  Dify内判定    │    │  Dify → MIL   │
│  → スコア算出  │ →  │  データ送信    │ →  │  ロジック      │ →  │  データ返却    │
│ 【MIL側】     │    │ 【API連携】   │    │ 【中西Dify】   │    │ 【API連携】   │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

### 三社の役割

| 会社 | 役割 |
|------|------|
| **MIL** | 学生向けのインタラクティブ動画・適性検査の画面を提供。学生の回答を集め、Dify APIにデータを送り、返ってきた結果を画面に表示する |
| **N1（エヌイチ）** | 中西製作所のDifyアカウント上に判定ロジック（ワークフロー）を構築・保守する。ロジックの設計・実装担当 |
| **中西製作所** | Difyアカウントの所有者。企業マスタ（推薦企業リスト）・判定条件の最終決定権を持つ |

### Difyアカウントについて

判定ロジックが動くDifyのアカウントは**中西製作所のもの**。
N1はこのアカウント上でワークフローを構築・運用するが、アカウント自体は中西が保有している。
したがって、**APIキーもエンドポイントURLも中西のDifyアカウントから発行される**。

## フェーズ1: 適性検査 → スコア算出【MIL側の処理】

### 目的

学生に51問の質問を出し、その回答から**「この学生はどんなタイプか」を数値で表す**。

### 質問票の構造（全51問）

| 設問 | 分類 | 何を測るか | 回答形式 | 算出されるスコア |
|------|------|-----------|---------|----------------|
| Q1〜Q24 | RIASEC | 仕事の志向性（6タイプ） | 各問1〜5点 | R, I, A, S, E, C（各4〜20点） |
| Q25〜Q36 | ポータブルスキル | 行動特性・コンピテンシー | 各問1〜5点 | ※今回の判定ではQ30・Q31のみ使用 |
| Q37〜Q48 | キャリア・アンカー | 価値観・カルチャー | 各問1〜5点 | ※今回の判定では直接使わない |
| Q49 | L尺度 | 嘘回答の傾向 | 1〜5点 | Q49（5ならスコア補正が発動） |
| Q50 | BtoB志向 | BtoB企業が好きか | 1〜5点 | Q50（3以上→BtoB寄り） |
| Q51 | メーカー志向 | インフラ/メーカーが好きか | 1〜5点 | Q51（中西との親和性判定用） |

### RIASECスコアの算出ロジック

Q1〜Q24の24問は、6タイプに4問ずつ割り振られている。各問は1〜5点で回答し、同じタイプの4問を合計する。

```
R型スコア = Q1 + Q2 + Q3 + Q4       （最低4点〜最高20点）
I型スコア = Q5 + Q6 + Q7 + Q8
A型スコア = Q9 + Q10 + Q11 + Q12
S型スコア = Q13 + Q14 + Q15 + Q16
E型スコア = Q17 + Q18 + Q19 + Q20
C型スコア = Q21 + Q22 + Q23 + Q24
```

**スコアが高い ＝ そのタイプの志向が強い**。

### 6タイプ（RIASEC）の意味

| コード | タイプ名 | 高スコアの人の特徴 | 合う仕事のイメージ |
|--------|---------|-------------------|-------------------|
| R | 現実型 | 手を動かすのが好き、技術志向 | エンジニア、製造、建設 |
| I | 研究型 | 分析・探究が好き、データ志向 | 研究開発、コンサル、分析 |
| A | 芸術型 | 創造・表現が好き、自由志向 | デザイナー、企画、クリエイティブ |
| S | 社会型 | 人の役に立ちたい、対人志向 | 教育、福祉、人事、カスタマーサポート |
| E | 企業型 | リーダーシップ、経営志向 | 営業、マネジメント、起業 |
| C | 慣習型 | 正確さ・管理が好き、堅実志向 | 経理、法務、事務、品質管理 |

### 回答結果の決まり方（上位タイプの判定）

6タイプのスコアを高い順に並べて、**1位（最もスコアが高いタイプ）と2位**を取る。この1位・2位が、その学生の志向性を表す。

**具体例**:

| 学生 | R | I | A | S | E | C | → 1位 | → 2位 | 解釈 |
|------|---|---|---|---|---|---|--------|--------|------|
| Aさん | 8 | 6 | 7 | 18 | 10 | 5 | **S型（18点）** | **E型（10点）** | 対人支援が最も強く、リーダーシップもある |
| Bさん | 10 | 8 | 16 | 12 | 19 | 7 | **E型（19点）** | **A型（16点）** | 経営志向が最も強く、創造性もある |
| Cさん | 18 | 15 | 6 | 8 | 7 | 12 | **R型（18点）** | **I型（15点）** | ものづくり志向が最も強く、研究志向もある |

**ポイント**:
- **1位タイプ**がメインの推薦企業を決める
- **2位タイプ**が「あなたにはこんな側面も」という補足の推薦企業を決める
- 1位と2位が同点の場合は、先にくるタイプ（R→I→A→S→E→Cの順）が1位になる

### このフェーズのアウトプット

51問の回答から、以下の**11項目の数値**が出る。これがフェーズ2でDifyに送られる。

| 項目 | 値の例 | 説明 | 何に使われるか |
|------|--------|------|-------------|
| R | 8 | R型スコア（4〜20） | 1位・2位タイプの判定 |
| I | 6 | I型スコア（4〜20） | 同上 |
| A | 7 | A型スコア（4〜20） | 同上 |
| S | 18 | S型スコア（4〜20） | 同上 |
| E | 10 | E型スコア（4〜20） | 同上 |
| C | 5 | C型スコア（4〜20） | 同上 |
| Q30 | 4 | 主体性（1〜5） | Step 2: 行動特性判定 |
| Q31 | 4 | 変革意欲（1〜5） | Step 2: 行動特性判定 |
| Q49 | 3 | 嘘回答検知（1〜5） | Step 0: L尺度補正 |
| Q50 | 4 | BtoB志向（1〜5） | Step 3,4: 企業リスト切替 |
| Q51 | 4 | メーカー志向（1〜5） | Step 3: 中西親和性判定 |

> [!question] 会議で確認すべきポイント
> - **RIASECの合計点算出はMIL側で行うか？** 生の回答データ（Q1〜Q24の各値）を送るのか、集計済みスコア（R=18等）を送るのか
> - **L尺度補正（×0.8）はどちら側で処理するか？** MIL側で補正してから送るか、Dify側で受け取ってから補正するか

## フェーズ2: MIL → Dify へのデータ送信【API連携】

### 通信の仕組み

中西製作所のDifyアカウント上にN1が判定ロジック（ワークフロー）を構築しており、そこにAPIが用意されている。MILのサーバーからこのAPIにデータを送ると、自動的に判定が走って結果が返ってくる。

**フェーズ2（送信）とフェーズ4（返却）は1回のHTTP通信の行きと帰り。**
MILがデータを送ったら、その応答としてDifyが結果を返す。電話でいうと、MILがかけて、Difyが電話口で答えるイメージ。

```
┌──────────────┐                    ┌───────────────────────────┐
│ MILのサーバー  │                    │ 中西製作所のDifyアカウント    │
│              │                    │ （N1がロジックを構築・運用）  │
│              │                    │                           │
│  ①データ送信  │ ── HTTPリクエスト →  │  ②判定ロジックが動く        │
│  （JSON +    │   （APIキー付き）    │  （フェーズ3の処理）        │
│   APIキー）  │                    │                           │
│              │                    │                           │
│  ④結果を受取  │ ← HTTPレスポンス ── │  ③結果をJSONで返す         │
│              │   （JSON）          │  （フェーズ4）             │
└──────────────┘                    └───────────────────────────┘
```

### APIキーとエンドポイントURL

APIキーとは、**「このAPIを使っていいですよ」という認証のための合鍵**のようなもの。リクエストを送るときに一緒に渡すことで、許可された相手からの通信だと確認される。

| 項目 | 何か | 誰が持っているか | 誰に渡すか |
|------|------|---------------|-----------|
| **エンドポイントURL** | データの送り先アドレス | 中西のDifyアカウントから発行される | MILに共有する |
| **APIキー** | 認証用の合鍵 | 中西のDifyアカウントから発行される | MILに共有する |

**流れ**:
1. 中西のDifyアカウント上でN1がワークフローを構築する
2. Difyの管理画面で「APIを公開」すると、**エンドポイントURL**と**APIキー**が発行される
3. この2つをMILに共有する（N1が代行して共有する想定）
4. MILは受け取ったURLとAPIキーを自社サーバーに設定し、学生が検査完了するたびにリクエストを送る

**MIL側が何かを発行したり、N1やMILのAPIキーを中西に登録する必要はない。**
中西のDifyアカウントから出た鍵を、MILが使う ── この一方向だけ。

### 送信データ（JSON形式）

MILのサーバーがDify APIに送る内容。フェーズ1で算出した11項目の数値。

```json
{
  "R":  8,
  "I":  6,
  "A":  7,
  "S":  18,
  "E":  10,
  "C":  5,
  "Q30": 4,
  "Q31": 4,
  "Q49": 3,
  "Q50": 4,
  "Q51": 4
}
```

全11項目、すべて数値。

### リクエストの技術仕様

```
POST https://api.dify.ai/v1/workflows/run
Authorization: Bearer {中西のDifyアカウントから発行されたAPIキー}
Content-Type: application/json

{
  "inputs": {
    "R": 8, "I": 6, "A": 7, "S": 18, "E": 10, "C": 5,
    "Q30": 4, "Q31": 4, "Q49": 3, "Q50": 4, "Q51": 4
  },
  "response_mode": "blocking",
  "user": "mil-server"
}
```

> [!question] 会議で確認すべきポイント
> - **JSONのキー名**は上記（R, I, A, S, E, C, Q30, Q31, Q49, Q50, Q51）で問題ないか
> - **各スコアの値域**の認識合わせ（RIASECは4〜20、Q30等は1〜5）
> - **APIキーの共有方法**と**共有タイミング**（Dify Professionalプラン切替後に発行）
> - **エンドポイントURLの確定タイミング**

## フェーズ3: Dify内の判定ロジック【中西Dify / N1構築】

### 目的

MILから受け取った11項目のスコアをもとに、**「この学生にどの企業を推薦するか」を決定する**。

### 判定に使うもの

| # | 何を | どこにあるか |
|---|------|-----------|
| 1 | MILから受け取った11項目のスコア | フェーズ2のリクエストで受信 |
| 2 | 企業マスタ（7タイプ × BtoB/BtoC = 14パターンの企業リスト） | 中西のDify内に事前登録済み |
| 3 | 推薦理由テキスト（各企業に対する「なぜあなたに合うか」の文章） | 中西のDify内に事前登録済み |

### 判定の全体像

判定は**4段階の優先順位**で上から順に判定する。あるステップに該当したら、そこで確定。下のステップには進まない。

| 順番 | ステップ | 判定内容 | 該当した場合の結果 |
|------|---------|---------|-------------------|
| 前処理 | Step 0: L尺度補正 | 嘘回答の傾向があればスコアを割り引く | （結果は出ない。スコアを補正するだけ） |
| 最優先 | Step 1: 未分化判定 | 全タイプのスコアに差がない | → **トモキャリ誘導**（企業推薦しない） |
| 2番目 | Step 2: 行動特性判定 | 主体性・変革意欲が高い | → **中西製作所を推薦** |
| 3番目 | Step 3: タイプ×志向判定 | 中西と親和性の高いタイプ＋業界志向 | → **中西製作所を推薦** |
| 最後 | Step 4: 通常判定 | 上記すべてに該当しない | → **タイプ別に企業を推薦** |

### Step 0: L尺度補正（前処理）

**何をするか**: 嘘回答の傾向がある学生のスコアを割り引く。結果は出さず、以降のStep 1〜4で使うスコアを補正するだけ。

| 項目 | 内容 |
|------|------|
| 判定条件 | Q49 = 5（最大値）のときだけ発動 |
| やること | R, I, A, S, E, C の各スコアに × 0.8 をかける |
| Q49が5以外 | 何もしない（スコアそのまま） |

**具体例**:

| 学生 | Q49 | 補正前 E | 補正前 R | → 補正後 E | → 補正後 R |
|------|-----|---------|---------|-----------|-----------|
| Xさん | **5** | 20 | 15 | **16**（×0.8） | **12**（×0.8） |
| Yさん | 3 | 20 | 15 | **20**（そのまま） | **15**（そのまま） |

### Step 1: 未分化判定（最優先）

**何をするか**: 6タイプのスコアに差がほとんどない学生を検出する。差がないということは志向が定まっていないので、企業を推薦しても意味がない。

| 項目 | 内容 |
|------|------|
| 判定条件 | 6タイプの最高点 − 最低点 ≦ 1.0 |
| 該当した場合 | → **トモキャリ誘導**（企業推薦せず、キャリア相談へ案内） |
| 該当しない場合 | → Step 2へ進む |

**具体例**:

| 学生 | R | I | A | S | E | C | 最高−最低 | 判定 |
|------|---|---|---|---|---|---|----------|------|
| Aさん | 12 | 12 | 11 | 12 | 12 | 11 | 12−11=**1.0** | → **トモキャリ誘導**（差がなさすぎる） |
| Bさん | 15 | 12 | 8 | 18 | 10 | 6 | 18−6=**12.0** | → Step 2へ（明確な差がある） |

### Step 2: 行動特性判定

**何をするか**: 主体性と変革意欲が高い学生を**中西製作所向き**と判定する。

| 項目 | 内容 |
|------|------|
| 判定条件 | Q30（主体性）≧ 4 **かつ** Q31（変革意欲）≧ 4 |
| 該当した場合 | → **中西製作所を筆頭に推薦** |
| 該当しない場合 | → Step 3へ進む |

**なぜこの条件か**: 中西製作所は「安定基盤がありながらベンチャー精神で変革を推進する」社風。主体性・変革意欲が両方高い学生はこの社風と合致する。

**具体例**:

| 学生 | Q30（主体性） | Q31（変革意欲） | 判定 |
|------|-------------|---------------|------|
| Aさん | 4 | 5 | → **中西製作所**（両方4以上を満たす） |
| Bさん | 5 | 3 | → Step 3へ（Q31が4未満） |
| Cさん | 3 | 4 | → Step 3へ（Q30が4未満） |
| Dさん | 2 | 2 | → Step 3へ（両方4未満） |

### Step 3: タイプ × 志向判定

**何をするか**: ①中西製作所と親和性の高いタイプ（E/I/A）に該当し、かつ②BtoB・メーカー志向が強い学生を検出する。2つの条件を両方満たした場合のみ中西と判定。

**条件①: 志向性タイプの判定**

| 条件 | 説明 |
|------|------|
| 1位タイプ or 2位タイプのどちらかが **E型、I型、A型** のいずれか | E型（経営・リーダーシップ）、I型（研究・分析）、A型（創造・企画）は中西が求める人材像と親和性が高い。1位でなくても、2位に含まれていればOK |

**条件②: 業界志向の判定**

| 条件 | 説明 |
|------|------|
| (Q50 + Q51) / 2 ≧ 3.0 | BtoB志向（Q50）とメーカー志向（Q51）の平均が3以上 ＝ 業界としても中西と合っている |

**具体例**:

| 学生 | 1位 | 2位 | Q50 | Q51 | 平均 | 条件① | 条件② | 判定 |
|------|-----|-----|-----|-----|------|-------|-------|------|
| Aさん | E(19) | A(16) | 4 | 4 | 4.0 | ✅ 1位がE | ✅ 4.0≧3.0 | → **中西製作所** |
| Bさん | S(18) | I(15) | 4 | 3 | 3.5 | ✅ 2位がI | ✅ 3.5≧3.0 | → **中西製作所** |
| Cさん | S(18) | R(15) | 4 | 4 | 4.0 | ❌ S,RにE/I/Aなし | ✅ 4.0≧3.0 | → **Step 4へ**（条件①が×） |
| Dさん | E(19) | A(16) | 2 | 2 | 2.0 | ✅ 1位がE | ❌ 2.0<3.0 | → **Step 4へ**（条件②が×） |

### Step 4: 通常判定（上記すべてに該当しなかった場合）

**何をするか**: 1位タイプ・2位タイプに対応する企業リストを、BtoB/BtoCの志向で切り替えて出す。

**判定の流れ**:

```
1位タイプを特定（例: S型）
    ↓
Q50でBtoB/BtoCを判定
  Q50 ≧ 3.0 → 企業マスタのBtoB列を使う
  Q50 < 3.0 → 企業マスタのBtoC列を使う
    ↓
企業マスタから「S型 × BtoB」の企業リスト → メイン推薦企業

2位タイプを特定（例: R型）
    ↓
同じQ50でBtoB/BtoCを判定
    ↓
企業マスタから「R型 × BtoB」の企業リスト → 補足推薦企業
```

**具体例**: 1位=S型、2位=R型、Q50=4 の場合

| | タイプ | Q50の判定 | 企業マスタの参照先 | 推薦される企業 |
|---|--------|----------|-----------------|-------------|
| メイン | S型 | 4 ≧ 3.0 → **BtoB** | S型 × BtoB | ベネッセ, リクルートマネジメント, パソナ, ワークスアプリ, LITALICO |
| 補足 | R型 | 4 ≧ 3.0 → **BtoB** | R型 × BtoB | 三菱電機, 村田製作所, 安川電機, 栗田工業, THK |

### 企業マスタ（中西のDify内に事前登録済み）

各Stepの判定結果に応じて、このマスタから企業リストを引く。

| タイプ | BtoB企業（Q50 ≧ 3.0のとき） | BtoC企業（Q50 < 3.0のとき） |
|--------|--------------------------|--------------------------|
| **中西型** | 中西製作所, キーエンス, 平田機工, マクニカ, ダイフク | ソニー, リクルート, メルカリ, カヤック, マネーフォワード |
| **R型** | 三菱電機, 村田製作所, 安川電機, 栗田工業, THK | トヨタ, 任天堂, スノーピーク, ヤマハ, マキタ |
| **I型** | 野村総研, 島津製作所, 中外製薬, オムロン, ユーザーベース | Google, 武田薬品, コーセー, エーザイ, メルカリ |
| **A型** | 電通, 博報堂, チームラボ, Sansan, 乃村工藝社 | サンリオ, オリエンタルランド, 資生堂, 集英社, 無印良品 |

## フェーズ4: Dify → MIL へのデータ返却【API連携】

### 通信の仕組み

フェーズ2でMILがリクエストを送った、**その返答（レスポンス）としてDifyが自動的にJSONを返す**。フェーズ2とフェーズ4は1回のHTTP通信の行きと帰りなので、フェーズ4のためにMIL側が別途何かを設定する必要はない。

- **APIキーは不要**（フェーズ2のリクエストに対する返答なので、認証は送信時に済んでいる）
- MIL側は、リクエストのレスポンスとしてJSONを受け取るだけ

### レスポンスJSON（想定）

```json
{
  "result_type": "nakanishi",
  "primary_type": "S",
  "secondary_type": "R",
  "primary_companies": ["中西製作所", "キーエンス", "平田機工", "マクニカ", "ダイフク"],
  "primary_reasons": [
    "中西製作所: 安定基盤×ベンチャー志向で…",
    "キーエンス: 技術力×営業力の…",
    "..."
  ],
  "secondary_companies": [],
  "secondary_reasons": [],
  "message": "あなたの主体性と変革意欲は、安定基盤×ベンチャー志向の企業で大きく花開きます。"
}
```

### 各項目の意味

| 項目 | 何が入るか | MIL側での使い方 |
|------|----------|---------------|
| result_type | 判定カテゴリ（`nakanishi` / `type_match` / `tomocari`） | 結果画面の出し分けに使う |
| primary_type | 1位の志向性タイプ（`R`〜`C`） | 「あなたは〇〇型」として表示 |
| secondary_type | 2位の志向性タイプ | 「〇〇の側面も」として補足表示 |
| primary_companies | メインの推薦企業リスト | 結果画面に企業名を表示 |
| primary_reasons | メイン推薦の理由テキスト | 各企業名の下に理由を表示 |
| secondary_companies | 補足の推薦企業リスト | 通常判定時のみ。中西該当・未分化時は空 |
| secondary_reasons | 補足推薦の理由テキスト | 同上 |
| message | 学生へのメッセージ | 結果画面の冒頭 or 末尾に表示 |

### 判定結果ごとのレスポンスの違い

| | nakanishi（中西該当） | type_match（通常判定） | tomocari（未分化） |
|---|---------------------|---------------------|------------------|
| result_type | `"nakanishi"` | `"type_match"` | `"tomocari"` |
| primary_companies | 中西＋同系統5社 | 1位タイプの企業5社 | 空 |
| secondary_companies | 空（中西への誘導を優先） | 2位タイプの企業5社 | 空 |
| message | 中西向けメッセージ | タイプ別メッセージ | トモキャリ誘導文 |

> [!question] 会議で確認すべきポイント
> - **レスポンスJSONのキー名・構造**はこれでMIL側の実装に合うか
> - **companiesは配列と文字列どちらが扱いやすいか**（上記は配列で記載）
> - **推薦理由のフォーマット**（企業名を含めるか、改行をどうするか等）
> - **レスポンスの処理時間**（現在の設計では0.1秒以内。MIL側のタイムアウト設定は？）
