# TOMOSHIKA ロジック整理（三者会議用）

> 適性検査の回答から企業推薦結果が表示されるまで、データがどう流れ・どう処理されるかを4フェーズで整理したもの。

---

## 全体フロー

```
┌──────────┐      ┌──────────┐      ┌──────────┐      ┌──────────┐
│ フェーズ1  │      │ フェーズ2  │      │ フェーズ3  │      │ フェーズ4  │
│ 適性検査   │ ──→ │ MIL→N1   │ ──→ │ N1内判定  │ ──→ │ N1→MIL   │
│ →スコア算出│      │ データ送信 │      │ ロジック   │      │ データ返却 │
│【MIL側】  │      │【MIL→N1】│      │【N1/Dify】│      │【N1→MIL】│
└──────────┘      └──────────┘      └──────────┘      └──────────┘
   学生が51問に       MILのサーバーが     受け取ったスコアで    判定結果を
   回答すると         N1のAPIに           企業マスタと         JSONで
   6タイプのスコア+    JSONで送信          付き合わせ           MILに返す
   特殊設問の値が出る
```

## フェーズ1: 適性検査 → スコア算出【MIL側の処理】

### 目的

学生に51問の質問を出し、その回答から**「この学生はどんなタイプか」を数値で表す**こと。

### 質問票の構造（全51問）

| 設問 | 分類 | 何を測るか | 回答形式 | 算出されるスコア |
|------|------|-----------|---------|----------------|
| Q1〜Q24 | RIASEC | 仕事の志向性（6タイプ） | 各問1〜5点 | R, I, A, S, E, C（各0〜20点） |
| Q25〜Q36 | ポータブルスキル | 行動特性・コンピテンシー | 各問1〜5点 | ※今回の判定では直接使わない |
| Q37〜Q48 | キャリア・アンカー | 価値観・カルチャー | 各問1〜5点 | ※今回の判定では直接使わない |
| Q49 | L尺度 | 嘘回答の傾向 | 1〜5点 | Q49（5なら補正発動） |
| Q50 | BtoB志向 | BtoB企業が好きか | 1〜5点 | Q50（3以上ならBtoB寄り） |
| Q51 | メーカー志向 | インフラ/メーカーが好きか | 1〜5点 | Q51（中西との親和性判定用） |

### RIASECスコアの算出ロジック

Q1〜Q24の24問は、6タイプに4問ずつ割り振られている。

```
R型スコア = Q1 + Q2 + Q3 + Q4      （各1〜5点 → 合計 4〜20点）
I型スコア = Q5 + Q6 + Q7 + Q8
A型スコア = Q9 + Q10 + Q11 + Q12
S型スコア = Q13 + Q14 + Q15 + Q16
E型スコア = Q17 + Q18 + Q19 + Q20
C型スコア = Q21 + Q22 + Q23 + Q24
```

**スコアが高い＝そのタイプの志向が強い**。

#### 6タイプの意味

| コード | タイプ名 | 高スコアの人の特徴 | 合う仕事のイメージ |
|--------|---------|-------------------|-------------------|
| R | 現実型 | 手を動かすのが好き、技術志向 | エンジニア、製造、建設 |
| I | 研究型 | 分析・探究が好き、データ志向 | 研究開発、コンサル、分析 |
| A | 芸術型 | 創造・表現が好き、自由志向 | デザイナー、企画、クリエイティブ |
| S | 社会型 | 人の役に立ちたい、対人志向 | 教育、福祉、人事、カスタマーサポート |
| E | 企業型 | リーダーシップ、経営志向 | 営業、マネジメント、起業 |
| C | 慣習型 | 正確さ・管理が好き、堅実志向 | 経理、法務、事務、品質管理 |

### 上位タイプの決め方

6タイプのスコアを高い順に並べて、**1位と2位**を取る。

**具体例**:

| R | I | A | S | E | C | → 1位 | 2位 |
|---|---|---|---|---|---|--------|------|
| 15 | 12 | 8 | 18 | 10 | 6 | **S型（18点）** | **R型（15点）** |
| 10 | 8 | 16 | 12 | 19 | 7 | **E型（19点）** | **A型（16点）** |
| 14 | 13 | 13 | 14 | 13 | 13 | **R型（14点）** | **S型（14点）** ※同点は先の方 |

この**1位・2位のタイプ**と**Q30, Q31, Q49, Q50, Q51の値**が、フェーズ3の判定に使われる。

### このフェーズのアウトプット（フェーズ2で送る値）

| 項目 | 値の例 | 説明 |
|------|--------|------|
| R | 15 | R型スコア（0〜20） |
| I | 12 | I型スコア（0〜20） |
| A | 8 | A型スコア（0〜20） |
| S | 18 | S型スコア（0〜20） |
| E | 10 | E型スコア（0〜20） |
| C | 6 | C型スコア（0〜20） |
| Q30 | 4 | 行動特性①（主体性） |
| Q31 | 4 | 行動特性②（変革意欲） |
| Q49 | 3 | L尺度（嘘回答検知） |
| Q50 | 4 | BtoB志向（3以上→BtoB寄り） |
| Q51 | 4 | メーカー志向（中西親和性判定用） |

> [!question] 会議で確認すべきポイント
> - **RIASECの合計点算出はMIL側で行うか？** 生の回答データ（Q1〜Q24の各値）を送るのか、集計済みスコア（R=15等）を送るのか
> - **L尺度補正（×0.8）はどちら側で処理するか？** MIL側で補正してから送るか、N1側で受け取ってから補正するか

## フェーズ2: MIL → N1 へのデータ送信【MIL → N1】

### 通信の仕組み

N1側が **Dify** というツール上に判定ロジックを構築しており、そこにAPIが用意されている。
MILのサーバーから、このAPIに対してデータを送ると、自動で判定が走って結果が返ってくる。

```
┌──────────────┐                    ┌──────────────────────┐
│ MILのサーバー  │  ── HTTPリクエスト →  │ N1のDify API          │
│              │  （JSON + APIキー）   │ （判定ロジックが動く）   │
│              │  ← HTTPレスポンス ──  │                      │
│              │  （JSON）            │                      │
└──────────────┘                    └──────────────────────┘
```

**1回のHTTP通信で送信→判定→返却が完結する**（フェーズ2〜4は技術的には1往復）。

### APIキーと接続情報

| 項目 | 提供元 | 説明 |
|------|--------|------|
| APIエンドポイントURL | **N1が提供** | Dify上のワークフローのURL。MILはここにデータを送る |
| APIキー | **N1が発行してMILに共有** | Difyの管理画面でN1が発行する認証キー。MILがリクエスト時にヘッダーに含める |

- APIキーはN1（Dify）側で発行・管理するもの
- MILはN1から受け取ったAPIキーを使ってリクエストを送る
- MIL側が何かを発行する必要はない

### 送信データ（JSON形式）

```json
{
  "R":  15,
  "I":  12,
  "A":   8,
  "S":  18,
  "E":  10,
  "C":   6,
  "Q30":  4,
  "Q31":  4,
  "Q49":  3,
  "Q50":  4,
  "Q51":  4
}
```

全11項目、すべてNumber型。MILのサーバーからHTTPリクエストのbodyとして送る。

### リクエストの形式（技術仕様）

```
POST https://api.dify.ai/v1/workflows/run
Authorization: Bearer {N1が発行したAPIキー}
Content-Type: application/json

{
  "inputs": {
    "R": 15, "I": 12, "A": 8, "S": 18, "E": 10, "C": 6,
    "Q30": 4, "Q31": 4, "Q49": 3, "Q50": 4, "Q51": 4
  },
  "response_mode": "blocking",
  "user": "mil-server"
}
```

> [!question] 会議で確認すべきポイント
> - **JSONのキー名**は上記（R, I, A, S, E, C, Q30, Q31, Q49, Q50, Q51）で問題ないか
> - **各スコアの値域**の認識合わせ（RIASECは0〜20? Q30等は1〜5?）
> - **APIキーの共有方法**（メール? Slack? セキュリティ上の取り扱い）
> - **エンドポイントURLの共有タイミング**（Dify Professionalプラン切替後）

## フェーズ3: N1内の判定ロジック【N1 / Dify】

### 目的

MILから受け取った11項目のスコアをもとに、**「この学生にどの企業を推薦するか」を決定する**こと。

### 使うもの

1. **MILから受け取った11項目のスコア**（フェーズ2の入力）
2. **企業マスタ**（7タイプ × BtoB/BtoC = 14パターンの企業リスト。N1のDify内に事前登録済み）

### 判定の考え方

判定は**4段階の優先順位**で行う。上のステップに該当したら、そこで判定確定。下のステップには進まない。

```
Step 0: L尺度補正（スコアの前処理）
    ↓
Step 1: 未分化判定 → 該当したら「トモキャリ誘導」で確定
    ↓（該当しなければ）
Step 2: 行動特性判定 → 該当したら「中西製作所」で確定
    ↓（該当しなければ）
Step 3: タイプ×志向判定 → 該当したら「中西製作所」で確定
    ↓（該当しなければ）
Step 4: 通常判定 → タイプ別に企業を推薦
```

### Step 0: L尺度補正（前処理）

**何をするか**: 嘘回答の傾向がある学生のスコアを割り引く。

| 項目 | 内容 |
|------|------|
| 判定条件 | Q49 = 5（最大値）のときだけ発動 |
| やること | R, I, A, S, E, C の各スコアに × 0.8 をかける |
| Q49が5以外 | 何もしない（スコアそのまま） |

**例**: Q49 = 5 の場合
- E = 20点 → 20 × 0.8 = **16点** として以降の判定に使う
- R = 15点 → 15 × 0.8 = **12点** として以降の判定に使う

### Step 1: 未分化判定（最優先）

**何をするか**: 6タイプのスコアに差がほとんどない（＝志向が定まっていない）学生を検出する。

| 項目 | 内容 |
|------|------|
| 判定条件 | 6タイプの最高点 − 最低点 ≦ 1.0 |
| 該当した場合 | **トモキャリ誘導**（企業推薦せず、キャリア相談へ案内） |
| 該当しない場合 | Step 2へ進む |

**具体例**:

| R | I | A | S | E | C | 最高-最低 | 判定 |
|---|---|---|---|---|---|----------|------|
| 12 | 12 | 11 | 12 | 12 | 11 | 12-11=**1.0** | → **トモキャリ誘導**（差がない） |
| 15 | 12 | 8 | 18 | 10 | 6 | 18-6=**12.0** | → Step 2へ（明確な差がある） |

### Step 2: 行動特性判定

**何をするか**: 主体性と変革意欲が高い学生を**中西製作所向き**と判定する。

| 項目 | 内容 |
|------|------|
| 判定条件 | Q30（主体性）≧ 4 **かつ** Q31（変革意欲）≧ 4 |
| 該当した場合 | **中西製作所を筆頭に推薦**（result_type = nakanishi） |
| 該当しない場合 | Step 3へ進む |

**なぜこの条件か**: 中西製作所は「安定基盤がありながらベンチャー精神で変革を推進する」社風。Q30とQ31が両方高い学生はこの社風と合致する。

**具体例**:

| Q30 | Q31 | 判定 |
|-----|-----|------|
| 4 | 5 | → **中西製作所**（両方4以上） |
| 5 | 3 | → Step 3へ（Q31が4未満） |
| 3 | 4 | → Step 3へ（Q30が4未満） |

### Step 3: タイプ × 志向判定

**何をするか**: 中西製作所と親和性の高いタイプ（E/I/A）で、かつBtoB・メーカー志向が強い学生を検出する。

| 項目 | 内容 |
|------|------|
| 条件① | 1位タイプ or 2位タイプのどちらかが **E型、I型、A型** のいずれか |
| 条件② | (Q50 + Q51) / 2 ≧ 3.0 |
| 両方を満たす場合 | **中西製作所を筆頭に推薦**（result_type = nakanishi） |
| どちらか欠ける場合 | Step 4へ進む |

**条件①の考え方**: E型（経営・リーダーシップ）、I型（研究・分析）、A型（創造・企画）は中西製作所が求める人材像と親和性が高い。1位でなくても2位に含まれていればOK。

**条件②の考え方**: BtoB志向（Q50）とメーカー志向（Q51）の平均が3以上 ＝ 業界志向としても中西と合っている。

**具体例**:

| 1位 | 2位 | Q50 | Q51 | 平均 | 判定 |
|-----|-----|-----|-----|------|------|
| E(19) | A(16) | 4 | 4 | 4.0 | → **中西製作所**（1位E ＋ 平均4.0） |
| S(18) | I(15) | 4 | 3 | 3.5 | → **中西製作所**（2位I ＋ 平均3.5） |
| S(18) | R(15) | 4 | 4 | 4.0 | → Step 4へ（1位S/2位RにE/I/Aなし） |
| E(19) | A(16) | 2 | 2 | 2.0 | → Step 4へ（平均3.0未満） |

### Step 4: 通常判定（上記すべてに該当しなかった場合）

**何をするか**: 1位タイプ・2位タイプそれぞれに対応する企業リストを、BtoB/BtoCの志向で切り替えて出す。

#### 判定の流れ

```
1位タイプを特定（例: S型）
    ↓
Q50でBtoB/BtoCを判定
  Q50 ≧ 3.0 → BtoBリストを選択
  Q50 < 3.0 → BtoCリストを選択
    ↓
1位タイプ × BtoB/BtoC → メイン推薦企業（primary_companies）

2位タイプを特定（例: R型）
    ↓
同じQ50でBtoB/BtoCを判定
    ↓
2位タイプ × BtoB/BtoC → 補足推薦企業（secondary_companies）
```

**具体例**: 1位=S型、2位=R型、Q50=4（BtoB寄り）の場合

| | タイプ | Q50 | 選ばれるリスト | 推薦企業 |
|---|--------|-----|-------------|---------|
| メイン | S型 | 4（≧3.0→BtoB） | S型 × BtoB | ベネッセ, リクルートマネジメント, パソナ, ワークスアプリ, LITALICO |
| 補足 | R型 | 4（≧3.0→BtoB） | R型 × BtoB | 三菱電機, 村田製作所, 安川電機, 栗田工業, THK |

### 企業マスタ（N1のDify内に事前登録済み）

判定結果とこのマスタを付き合わせて、推薦企業を決定する。

| タイプ | BtoB企業（Q50 ≧ 3.0） | BtoC企業（Q50 < 3.0） |
|--------|----------------------|----------------------|
| **中西型** | 中西製作所, キーエンス, 平田機工, マクニカ, ダイフク | ソニー, リクルート, メルカリ, カヤック, マネーフォワード |
| **R型** | 三菱電機, 村田製作所, 安川電機, 栗田工業, THK | トヨタ, 任天堂, スノーピーク, ヤマハ, マキタ |
| **I型** | 野村総研, 島津製作所, 中外製薬, オムロン, ユーザーベース | Google, 武田薬品, コーセー, エーザイ, メルカリ |
| **A型** | 電通, 博報堂, チームラボ, Sansan, 乃村工藝社 | サンリオ, オリエンタルランド, 資生堂, 集英社, 無印良品 |
| **S型** | ベネッセ, リクルートマネジメント, パソナ, ワークスアプリ, LITALICO | 日本赤十字, 損保ジャパン, 星野リゾート, Zoff, クラシコム |
| **E型** | 三菱商事, 伊藤忠, 光通信, サイバーエージェント, レバレジーズ | ソフトバンク, 楽天, ファーストリテイリング, ZOZO |
| **C型** | 日本生命, 東京海上, 三菱UFJ, 富士通, オービック | JR東日本, ANA, 日本郵便, 公務員, ゆうちょ銀行 |

#### 付き合わせのまとめ

| 判定結果 | どの企業リストを使うか | 具体例 |
|---------|---------------------|--------|
| Step 2 or 3で**中西該当** | 「中西型」の行を使う。Q50でBtoB/BtoC切替 | Q50=4 → 中西製作所, キーエンス, 平田機工… |
| Step 4で**通常判定** | 1位タイプの行（メイン）＋ 2位タイプの行（補足）を使う。それぞれQ50で切替 | 1位E×BtoB → 三菱商事, 伊藤忠… / 2位A×BtoB → 電通, 博報堂… |
| Step 1で**未分化** | 企業マスタは使わない | トモキャリ誘導 |

> [!question] 会議で確認すべきポイント
> - 企業マスタの内容に変更・追加はないか（津丸さん確認）
> - 中西該当時にBtoCリスト（ソニー, リクルート等）を出すケースがありうるが、それでよいか
> - 推薦理由テキストの内容・トーンは誰が決めるか

## フェーズ4: N1 → MIL へのデータ返却【N1 → MIL】

### 通信の仕組み

フェーズ2でMILがリクエストを送った**その返答（レスポンス）**として、N1のDifyが自動的にJSONを返す。MIL側が別途何かを受け取る設定をする必要はない。

```
フェーズ2（リクエスト）とフェーズ4（レスポンス）は1往復のHTTP通信

MILのサーバー  ── リクエスト（11項目のスコア）──→  N1 Dify API
MILのサーバー  ←── レスポンス（判定結果JSON） ──  N1 Dify API
```

- **APIキーは不要**（フェーズ2のリクエストに対する返答なので、認証は送信時に済んでいる）
- MIL側は、リクエストのレスポンスとしてJSONを受け取るだけ

### レスポンスJSON（想定）

```json
{
  "result_type": "nakanishi",
  "primary_type": "S",
  "secondary_type": "R",
  "primary_companies": ["中西製作所", "キーエンス", "平田機工", "マクニカ", "ダイフク"],
  "primary_reasons": [
    "中西製作所: 安定基盤×ベンチャー志向で…",
    "キーエンス: 技術力×営業力の…",
    "..."
  ],
  "secondary_companies": [],
  "secondary_reasons": [],
  "message": "あなたの主体性と変革意欲は、安定基盤×ベンチャー志向の企業で大きく花開きます。"
}
```

### 各項目の意味と、判定結果による出し分け

| 項目 | 内容 | nakanishi の場合 | type_match の場合 | tomocari の場合 |
|------|------|-----------------|------------------|----------------|
| result_type | 判定カテゴリ | `"nakanishi"` | `"type_match"` | `"tomocari"` |
| primary_type | 1位タイプ | S, E 等 | S, E 等 | S, E 等 |
| secondary_type | 2位タイプ | R, I 等 | R, I 等 | R, I 等 |
| primary_companies | メイン推薦企業 | 中西＋同系統5社 | 1位タイプの企業5社 | 空（企業推薦なし） |
| primary_reasons | メイン推薦理由 | 各社の理由テキスト | 各社の理由テキスト | 空 |
| secondary_companies | 補足推薦企業 | 空（中西への誘導を優先） | 2位タイプの企業5社 | 空 |
| secondary_reasons | 補足推薦理由 | 空 | 各社の理由テキスト | 空 |
| message | 学生へのメッセージ | 中西向けメッセージ | タイプ別メッセージ | トモキャリ誘導文 |

### MIL側での利用想定

MILはこのJSONを受け取り、結果画面（インタラクティブ動画上）に表示する。
- `result_type` で画面の出し分け
- `primary_companies` + `primary_reasons` でメインの企業推薦を表示
- `secondary_companies` があれば「こんな側面も」として補足表示
