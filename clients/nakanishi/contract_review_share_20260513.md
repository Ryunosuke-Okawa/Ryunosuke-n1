- **契約書は未署名**なので、今のうちに条文を修正可能
- 方針：**期日で切る**のではなく、**「7割納品」のスコープ定義**で線引きする
- キャッシュフロー：着手金50万円＝5月末入金、残金50万円＝6月末入金見込（影響なし）

## 2. 何の齟齬が生まれているか

### 当社の認識
- **契約範囲は納品（検収完了）まで**
- 検収完了後の修正・保守は **別契約・別料金**
- 7割→10割への追加開発は **別契約**

### 先方（中西製作所）の認識
- **契約範囲は仮納品〜検収〜保守運用（9月末まで）すべて**
- 検収完了後の **軽微な修正は委託料に含まれる**
- 7割→10割への追加開発は **別契約**（ここだけ一致）

### 比較表

| 観点 | 当社 | 先方 | 一致／不一致 |
|------|------|------|-----------|
| 契約範囲の終点 | 検収完了まで | 9月末の保守運用まで | **不一致** |
| 検収後の軽微修正 | 別契約・別料金 | 委託料に含まれる | **不一致** |
| 7割→10割の追加開発 | 別契約 | 別契約 | 一致 |

→ **「保守運用が契約内か外か」が齟齬の核心**。

---

## 3. 齟齬が生まれた原因：契約書内の文言矛盾

第2条のなかで、**正反対の文言が並存**している。

### A) 先方解釈を支える条文

**第2条・業務範囲⑤**：
> テスト運用開始後（2026年5月〜）の改善・保守支援（本番環境最終納期：2026年9月末）

**第2条・完了条件**：
> 2026年4月末の仮納品後、2026年5月から同年9月末までの間、テスト運用を通じて軽微な修正・改善を実施するものとし、当該修正・改善のうち、本契約締結時に予定されていた仕様の範囲内のものについては、本契約の委託料に含まれるものとする。

### B) 当社解釈を支える条文

**第2条・業務スケジュール**：
> 2026年5月以降：ブラッシュアップ・保守運用（**別途見積・別契約**にて対応）

**同上**：
> 乙は、検収完了日から3か月間、成果物に関する軽微な不具合修正および仕様書記載内容の範囲内の改善対応を **無償**で行うものとする

### 矛盾の構造

- 業務範囲⑤＋完了条件　＝「5月〜9月末まで仕様範囲内の修正は委託料内」
- 業務スケジュール　　　＝「5月以降は別途見積・別契約」「検収完了から3か月のみ無償」

→ **本文（業務範囲・完了条件）と進行表（スケジュール）が逆の主張**。トラブル時は本文側が強く読まれて、**先方有利**になりやすい。

## 4. 現状（このまま署名）のリスク

| # | リスク | 影響度 |
|---|-------|------|
| 1 | 検収完了後も「保守内」として **無償対応を9月末まで要求**される | 大：工数流出 |
| 2 | 「軽微な修正」の定義が曖昧なため、**先方が"軽微"と言えば通る**建付け | 大：判断基準なし |
| 3 | 検収完了通知の発火タイミングが先方マターのため、**残金50万円の入金が遅延**する可能性 | 中：キャッシュ |
| 4 | 後続の追加開発を別契約として提案しても「契約書上は保守内」と押し戻される | 中：売上機会逸失 |
| 5 | 保守期間中の対応で「これは保守内／外」の押し付け合いが発生 | 中：関係性悪化 |

→ **特に1と2は、放置すると半年間（5〜9月）にわたって工数を吸われ続ける構造**。

## 5. リスク回避のために整理すべきこと

### 5-1. 契約書の文言矛盾を解消する

第2条の業務範囲⑤・完了条件・業務スケジュールを **統合・整合化** する。

### 5-2. 「7割納品」のスコープを定義する

期日で切るのではなく、**成果物の中身**で線引き。

### 5-3. 保守運用の対象を **やる／やらない** の具体例で明文化する

「軽微」の解釈余地を消す。

## 6. 7割納品の定義（契約書 検収条件①〜④ベース）

| # | 機能 | 仕様 |
|---|------|------|
| ① | MIL適性検査回答データ（JSON）の受領・集計処理 | 全51問・6タイプ分類済みデータの受領 |
| ② | 推薦対象判定の分岐ロジック | 中西製作所への推薦可否判定 |
| ③ | 推薦対象向け：中西製作所への案内フロー | 単一企業への誘導 |
| ④ | 推薦対象外向け：タイプ別マッチング＋案内フロー | 企業DB＝**70社程度**ベース |

→ 「7割」とは、**この①〜④が仕様通り動作している状態**を指す。

---

## 7. 保守運用の線引き：やるもの／やらないもの

### 保守運用 内（委託料に含まれる・無償）

①〜④の **仕様の中での改善・維持** に限定。

| 区分 | 具体例 |
|------|-------|
| **不具合修正** | JSON受領処理のエラー／判定ロジックの誤判定／案内フロー不具合 |
| **精度向上・調整** | 判定ロジックの閾値調整、マッチングスコアの再キャリブレーション |
| **表記・UI微修正** | 案内文の言い回し変更、エラーメッセージ文言調整、見出し・改行などの体裁 |
| **既存出力の項目追加** | 内部で既に保持しているデータ（診断型など）をAPIレスポンスに含める |
| **企業DB差し替え** | 既存70社内の企業を別企業と差し替え（**月◯件まで**※要決定） |
| **API仕様書の更新** | 上記対応に伴うサンプル更新・仕様書追記 |

### 保守運用 外（別途見積・別契約）

①〜④の **仕様そのものを拡張する追加開発**。

| 区分 | 具体例 |
|------|-------|
| **企業DBの社数追加** | 70社 → 100社など、データベースの母数拡張 |
| **新規判定軸の追加** | 6タイプ以外の分類追加、新パラメータでの判定追加 |
| **新規分岐ロジック** | 年齢・性別など新しい条件で文言や案内を出し分ける |
| **新規案内フロー** | 新しい企業群への案内、新規連携先の追加 |
| **出力構造そのものの変更** | 1企業表示 → 3社比較表示など、レスポンス構造の根本変更 |
| **外部システム連携** | MIL以外の検査ベンダーとの連携、新規API連携 |
| **新規UI／管理画面** | 仕様書に記載のない新しい画面・操作機能 |

### 判断の原則

> **①〜④の "仕様の中" の改善＝保守内 ／ ①〜④の "仕様そのもの" の拡張＝別契約**

## 9. 提案する対応

### ステップ1：契約書の修正（未署名のうちに）

第2条を以下の構造で書き換え：

1. **業務範囲⑤** を「**検収条件①〜④の動作維持・改善に限定した保守運用**」と明記
2. **完了条件** に「保守期間中、委託料に含まれる対応 / 別契約となる対応」のリストを列挙（本資料 第7章ベース）
3. **業務スケジュール** の「5月以降は別途見積・別契約」記載と整合させる

### ステップ2：先方への説明資料を準備

「7割納品の定義」「保守内/外の線引き表」を一枚にまとめて、契約書改訂の理由とセットで説明。

### ステップ3：髙橋承認後、先方提示

中西製作所側でも社長承認が必要。スケジュールに余裕を持って動く。
