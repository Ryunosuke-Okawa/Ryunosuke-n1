# proposal-outline — 髙橋メソッド提案骨子設計（対話型）

商談の文字起こしから、髙橋メソッドに基づいて提案資料の骨子をステップバイステップで対話しながら設計し、最終的にGoogle Slidesとして出力するスキル。
`/auto-minutes` 完了後に続けて実行することも、単独で実行することも可能。

## 基本思想（髙橋メソッド）

提案資料は商談ストーリーの一部であり、資料単体で考えてはいけない。
資料を作る目的は二つだけ：

1. **具体の内容をわかりやすく伝えること**
2. **御社のために時間をかけて考えたという誠意を見せること**

この二つを満たすための原則：

- 資料でなければ伝わらない部分だけを資料にする。雑談・スタンス合わせ・信頼構築は口頭の領域であり、資料には入れない
- **20分で説明しきれるボリューム**を目安にする。長い資料は相手を飽きさせる
- 各セクションで必ず**合意を取る設計**にする。ずれたまま進む商談は最悪
- **事実と推察は絶対に混ぜない**。事実は事実、課題は課題、提案は提案。それぞれ明確に分ける
- **サービス名から入らない**。相手が知りたいのは「自分たちの課題がいつまでにどうなるか」であり、サービスの説明ではない

## 運用の鉄則

- **自動実行（ステップ0〜3）**: ファイル選択後、ステップ0→1→2→3は途中で承認を取らず**全自動で連続実行**する。骨子（ステップ3）の出力が完了した時点で**一度だけ**ユーザーに確認を取る
- **骨子OKの後**: ユーザーが承認したらステップ4（詳細化）→ ステップ5（PPTX生成）を**途中で承認を取らず全自動で連続実行**する。修正・変更指示があればその箇所を修正してから進む
- **事実と推察の分離**: 商談中の発言＝事実、こちらの解釈・仮説＝推察。出力時に必ずラベルをつける
- **ナレッジの活用**: サービス資料（上記）および過去の提案資料が `03_projects/` や `00_context/` にある場合、構成・表現の勝ちパターンを優先的に参照する
- **Kawaru Team 人数ロジック**: 1セッションの定員は**15〜25名**。「N名を研修したい」という要件があった場合、以下のルールで計算する：
  - 25名以下 → 1セッション
  - 26〜50名 → 25名 × 2セッション
  - 51〜75名 → 25名 × 3セッション（目安）
  - 費用計算は「15万円/名 × 人数」で算出。セッション数は実施スケジュール設計に使用する
  - スライドの「費用」「実施規模」「スケジュール」には必ずこのセッション分割ロジックを反映すること
- **PPTXカラーシステム（統一寒色4系統）**: 以下の色のみ使用。RED/GREEN/BLUE/ORANGEの鮮やか4色同時使用は禁止（トーン崩壊の原因）
  - `TEAL`（#42A4AF）: ブランド主役・推奨要素・解決方向
  - `SLATE`（#4A6C8C）: 第2アクセント・構造的要素・寒色サブ
  - `CORAL`（#C95454）: 課題・問題・Beforeカード
  - `SAGE`（#3A8F63）: 解決策・成果・Afterカード
  - `NAVY`（#1C3557）: 最終ステップ・強調・制約要素
  - 4段階ステップは必ず **TEAL→SLATE→SAGE→NAVY** の順で配色。3段階は TEAL→SLATE→SAGE
- **PPTXデザインルール**: 全スライドの背景は`#FFFFFF`（白）で統一。クロージング・締めスライドも含めて例外なし。暗い背景は使用しない
- **PPTXレイアウト安全ルール（フォント拡大時）**: スライドキャンバスは13.33"×7.50"。全要素のend_y（開始Y+高さ）が7.50"以内に収まることを必ず確認すること
  - `_FSZ`スケールマップで+2pt適用後、行高さ・カード高さも比例して拡大する
  - テーブル行: `row_h`を拡大する際、行数×(row_h+gap)の合計+cy+後続要素の高さが7.50"を超えないか計算する
  - グリッドレイアウト（2×2モジュール等）: 下段カードのend_y = cy + offset + card_h ≤ 7.40"
  - バッジ（num_badge）は右上に配置し、テキスト領域と重ならないようにする（左上だとタイトルと干渉する）
  - 固定offset（cy4+5.90等）は「カード端y + 小さな余白」で再計算し、end_y ≤ 7.50"にする

## ステップ0: 商談コンテキストの把握

文字起こし（or 議事録）が入力されたら、以下を整理して出力する。確認は取らず自動でステップ1へ進む。

**出力内容:**
- **商談の基本情報**: 誰と誰が、何回目の商談で、何について話したか
- **推奨ターゲット**: 次回の提案資料は誰向けに作るべきか（経営層 / 現場・推進担当者 / その他）。文脈から判断し、理由を添えて提案する
- **前回商談で合意していたネクストアクション**: 次回何をする約束だったか

→ **自動でステップ1へ進む。**

## ステップ1: 商談ストーリーの設計

次回の商談全体の流れを設計する。提案資料はこのストーリーの中の一部として位置づける。

**出力内容 — 次回商談の推奨ストーリー（時間配分の目安つき）:**

| フェーズ | 内容 | 手段 | 目安時間 |
|---------|------|------|--------|
| 1 アイスブレイク | 雑談・関係構築 | 口頭 | 3〜5分 |
| 2 スタンス合わせ | 全員が同じ方向を向く前提づくり | 口頭 | 2〜3分 |
| 3 前回の振り返り | 前回の商談概要の認識合わせ → 合意を取る | 資料 | 3〜5分 |
| 4 現状と課題の整理 | 事実ベースの現状 → 課題の特定 → 合意を取る | 資料 | 5〜7分 |
| 5 提案（ゴール→根拠→具体策） | 課題をいつまでにどうするか → サービス概要 | 資料 | 8〜10分 |
| 6 確認・ディスカッション | 提案とのズレ確認、質疑 | 口頭 | 5〜10分 |
| 7 ネクストアクション | 次のステップを合意 | 口頭 | 2〜3分 |

※ 上記はテンプレート。商談の文脈に合わせてフェーズの内容・時間配分をカスタマイズして出力する。

→ **自動でステップ2へ進む。**

## ステップ2: 内容整理（振り返り・現状・課題の分離）

資料に落とし込む前段階として、文字起こしの内容を以下の4層に整理する。
**A〜Dは明確に別物であり、絶対に混ぜない。**

### A. 前回の振り返り

前回の商談の概要を簡潔にまとめる。ここでは深掘りしない。

- 参加者（先方・自社）
- 話した内容の概要
- 先方が示した関心・反応
- 合意したネクストアクション

### B. 現状の整理（事実ベース）

先方の現在の状態を、商談での発言に基づいて**事実のみ**で整理する。

- **【事実】**として明記。推察や解釈は入れない
- 例：「AI推進室はあるが、具体的な施策は進んでいない」「リテラシーにばらつきがある」

### C. 課題の特定

現状の事実から、何が課題なのかを切り出す。

- 先方が課題と認識しているもの（商談中の発言から）→ **【事実】**
- こちらが課題と捉えるべきと考えるもの（推察・仮説）→ **【推察】** と明記
- 先方が課題と思っていない可能性もあるため、押しつけない表現にする

### D. 課題のロジックツリー（構造化）

Cで特定した課題を、上位概念→中位概念→下位概念の階層構造で整理する。

**構造ルール:**
- **上位概念（根本課題）**: 先方が解決したい本質的な問い。1〜2個に絞る
- **中位概念（要因）**: 根本課題を引き起こしている要因や領域。3〜4個
- **下位概念（具体事象）**: 商談で実際に発言・確認された具体的な事象・症状

**出力フォーマット（テキストツリー）:**

```
[根本課題]
├── [要因A]
│   ├── [具体事象A-1]【事実/推察】
│   └── [具体事象A-2]【事実/推察】
├── [要因B]
│   ├── [具体事象B-1]【事実/推察】
│   └── [具体事象B-2]【事実/推察】
└── [要因C]
    └── [具体事象C-1]【事実/推察】
```

このロジックツリーはステップ3の課題スライドに直接使用する。

→ **自動でステップ3へ進む。**

## ステップ3: 提案骨子（スライド構成）

スライド構成を設計する。このステップの出力が完了したら**ユーザーに確認を取る（唯一の承認ポイント）**。

### 1スライド1メッセージの原則（最重要）

**1枚のスライドで伝えることは1つだけ。** 複数のメッセージを詰め込まない。

- 「何を言いたいか」が1行で言えないなら、スライドを分ける
- 情報が多いと感じたら、それはスライドを増やすサインである
- **スライド枚数を増やすことを恐れない**。コンパクトにまとめようとして内容が薄くなる・わかりにくくなるほうが問題

### タイトルの書き方ルール

スライドタイトルは**短く・本質を突く**形式にする。長い文章はNG。

| NG（文章タイトル） | OK（短いタイトル） |
|---|---|
| 「エンジニアの約半数が中国語話者であり、日本語研修だけでは対応できない組織構造です」 | 「エンジニアの約半数が中国語話者」 |
| 「AI機能を組み込めなければ、保険SaaS市場での競争力を失うリスクがあります」 | 「AI非対応は、競争力喪失に直結する」 |
| 「ヒアリング2時間で御社専用のカリキュラムをゼロから設計します」 | 「ヒアリング2時間でカリキュラムをゼロ設計」 |

- タイトルは **15文字前後** を目安にする
- 「です・ます」で終わる説明文はタイトルではなく本文に入れる
- セクション名（「現状①」「課題③」など）はサブタイトル（小さい文字）として添えることで、タイトルを短くしつつ位置づけを明確にできる

### スライドストーリーの基本型

スライド構成は商談の文脈に応じて柔軟に変えてよい。枚数・順序・パート分けはあくまでガイドラインだが、**1スライド1メッセージを守ると自然に枚数が増える**ことを前提とした設計をする。

**目安スライド枚数：全体で15〜20枚**

---

### PPTXスライド順序ルール（必須）

生成するすべてのPPTXは以下の順序でスライドを配置する。変更不可。

```
1:  表紙
2:  目次（toc_slide関数で生成。全セクションを一覧表示）
3:  セクション区切り「01 振り返り」
4〜: 振り返りスライド群
?:  セクション区切り「02 現状」
〜: 現状スライド群
?:  セクション区切り「03 課題」
〜: 課題スライド群
?:  セクション区切り「04 ゴール」
〜: ゴールスライド群
?:  セクション区切り「05 提案詳細」（根拠・詳細・費用を含む）
〜: 提案詳細スライド群（根拠 → 詳細 → 費用の順）
?:  ネクストアクション
?:  締めスライド
?:  参考資料 区切り（ref_divider関数）
?:  会社概要（参考資料）
?:  導入実績（参考資料）
```

**重要ルール：**
- 会社概要・導入実績は**必ず末尾の参考資料として配置**（冒頭には置かない）
- セクション区切りは `section_divider(prs, 番号, タイトル, 説明)` 関数で生成
  - 番号は '01'〜'05'（2桁ゼロパディング）
  - 白背景・左TEALバー・装飾大文字＋セクション名の統一デザイン
- 参考資料区切りは `ref_divider(prs)` 関数で生成
- 目次は `toc_slide(prs, sections)` 関数で生成（sections = [(番号, タイトル, 説明), ...] のリスト）

**【パート1：振り返り】**（目安1〜2枚）
- 前回の商談概要・参加者・合意ネクストアクション
- 目的：全員の記憶を揃え、スタートラインに立たせる

**【パート2：現状】**（目安2〜4枚）

現状を「事業・組織概要」「AI活用の現状」「制約・環境」など**テーマごとに分けて1枚ずつ**設計する。1枚にまとめようとしない。

- 事業概要・組織構造（1枚）
- AI活用の現状（1枚）
- ツール・制度・環境の制約（1枚）

目的：事実を積み上げ、「確かにそういう状況です」という認識合意を取る

**【パート3：課題】**（目安3〜5枚）

課題も**要因ごとに1枚ずつ**設計する。全部を1枚に詰め込まない。

- 課題の全体マップ（1枚）：課題を俯瞰するロジックツリー
- 各要因の詳細（1〜3枚）：要因ごとに1枚ずつ掘り下げる
  - **ステップ2Dのロジックツリーをそのままスライドに落とし込む**
  - 事実・推察のラベルを色で区別する
- 緊急性・タイムライン（必要に応じて1枚）：なぜ今動くべきかを示す

目的：事実と課題を分離し、「この課題認識で合っていますか？」という認識の合意を取る

**【パート4：提案ゴール】**（目安2枚）
- 目指す姿・Before/After（1枚）：サービス名ではなく相手の未来像を先に出す
- 成果指標・KPI（1枚）：達成すべき状態を具体的に定義する

**【パート5：提案根拠】**（目安2〜3枚）
- 一般的なアプローチとの比較（1枚）：なぜこの提案でなければならないか
- 自社ならではの強み（1枚）：差別化ポイントを具体的に示す

**【パート6：提案詳細】**（目安3〜4枚）

提案の内容も**テーマごとに分けて1枚ずつ**設計する。

- 対象・内容（1枚）
- 実施形式・スケジュール（1枚）
- 費用・条件（1枚）
- 進め方・フロー（1枚）

**【パート7：クロージング】**（目安1〜2枚）
- ネクストアクション（必須）：STEPを図で示す
- 本日の確認事項（任意）：商談で合意を取るべき3点程度を提示
- 料金（任意：商談の流れに応じて入れるかどうか判断する）

### 提案内容に含めるべき2種類の要素

1. **前回の商談で実際に提案・言及した内容**（議事録から読み取れるもの）→ 必ず入れる
2. **AIによる別角度の提案アイデア**（推論ベース）→「こういう視点もあるかもしれません」のトーンで別途提示。ユーザーが判断して採用・不採用を決める

|------------|--------|-------------------------------|--------------------------|-------------|----------|
| 1 | 振り返り | 前回のお打合せ振り返り | 振り返り | テキスト＋サマリー表 | スタートラインを揃える |

## ステップ4: スライド詳細化

骨子の承認後、各スライドをそのまま書き込めるレベルまで詳細化する。**文字起こしの内容を最大限に引用・活用して肉付けすること。コンパクトにまとめようとしない。**

### 各スライドの出力フォーマット

```
---
### スライド[No.] ｜ セクション：[セクション名]
**タイトル**:（15文字前後・短く本質を突く。文章にしない）

**ボディコピー**:
（箇条書き・表・数値・引用で詳細を記述。文字起こしの発言は「」で引用する。
 事実は【事実】、推察は【推察】とラベルをつける。
 第三者が読んでも状況が理解できる粒度で書く）

**商談での使い方メモ**:
（口頭補足のポイント・合意を取るべき箇所・先方が質問しそな点）
---
```

**詳細化のポイント:**
- 文字起こしに出てきた**実際の発言**は積極的に引用する（説得力が増す）
- 数値・固有名詞・具体的な状況は必ず入れる（抽象的な記述はNG）
- 「商談での使い方メモ」は「このスライドで何の合意を取るか」を必ず書く
- 1スライドの詳細が長くなるのは良いこと。情報を削って薄くしない

→ **承認を取らず、自動でステップ5へ進む。**

## ステップ5: PPTX生成 → Google Drive アップロード

スライド詳細の承認後、python-pptx でデザイン込みのPPTXを生成し、Google Driveにアップロードする。

### 5-1. スライド全体構成

以下の順序でスライドを生成する：

| 順番 | スライド | 種別 |
|---|---|---|
| 1 | **表紙** | 固定 |
| 2 | **会社概要** | 固定 |
| 3 | **導入実績** | 固定 |
| 4〜N | **提案スライド**（ステップ4の各スライド） | 可変 |
| N+1 | **締めスライド** | 固定 |

### 5-2. 固定スライドの内容（毎回必ず同じ内容で生成する）

#### 表紙
- タイトル：「生成AI支援ご提案書」
- サブタイトル：「{先方会社名} 様向け」
- 左下：日付（小さく）
- 右下：「株式会社エヌイチ」

#### 会社概要
タイトル：「会社概要」

左側に以下の表を配置：

| 項目 | 内容 |
|------|------|
| 会社名 | 株式会社エヌイチ |
| 経営陣 | 代表取締役CEO 奥山幸生 / 取締役COO 髙橋悠 / 取締役CHRO 米澤浩明 |
| 資本金 | 1,000万円 |
| 設立 | 2023年10月 |
| 所在地 | 東京都新宿区西新宿3-7-30 フロンティアグラン西新宿12F |
| Mission | 新しい働く『カタチ』を創る |
| Vision | AI（アイ）ある社会に |

右側にサービス4つをカード形式で配置：
- **Kawaru**：業務自動化SaaS
- **Kawaru Team**：フルカスタマイズAI研修
- **Kawaru BPO**：AI業務効率化代行
- **Kawaru Coach**：AI顧問サービス

#### 導入実績
タイトル：「導入実績」
サブタイトル：「あらゆる規模の企業様に導入されています」

上段に導入企業名を横並びで配置：
株式会社ブロードリンク / 株式会社中西製作所 / 株式会社ミズカラ / 株式会社エル・ティー・エス リンク / FRUOR株式会社 / 株式会社BIRDY / 株式会社オンシナジー / キリングループ

下段に代表事例を3列のカード形式で配置（各カードは課題→成果の2行に収める）：

- **事例1：キリングループ様**
  - 課題：AI未経験、活用イメージなし
  - 成果：全員が「業務でAIを使いたい」と回答、満足度8.5点

- **事例2：株式会社ミズカラ様**
  - 課題：問い合わせ対応に月283時間
  - 成果：月85時間へ削減（約200時間＝25営業日分の削減）

- **事例3：株式会社BIRDY様**
  - 課題：業務の属人化、営業効率の壁
  - 成果：バックオフィス効率化、経営資源を成長領域にシフト

#### 締めスライド
- メイン：「私たちが提供するのは「ツール」や「研修」ではありません。貴社の未来を創る「AI人材」と、組織全体の「成長」です。」
- 下部：「株式会社エヌイチ」

### 5-3. デザインルール

**基本方針: スピード重視・わかりやすさ優先**
- デザインの凝り方よりも「内容が一瞬で伝わるか」を最優先にする
- 複雑なレイアウトより、シンプルで読みやすい構成を選ぶ
- 生成速度を落とす装飾・複雑な図形処理は避ける

**配色:**

| 用途 | カラーコード | 備考 |
|------|------------|------|
| **背景（全スライド共通）** | `#FFFFFF` | 純白。グレー下地は使わない |
| **アクセント（メイン）** | `#42A4AF` | タイトルライン・強調・バッジ・左アクセントバー |
| アクセントライト | `#EAF6F7` | ハイライトカード・成果・ポジティブ要素の背景 |
| カード背景 | `#F9FAFB` | カード型ブロックの背景（白に近いグレー） |
| 区切り線 | `#E5E7EB` | セクション区切り・フッター |
| 課題強調 | `#EF4444` | 課題・問題点の強調（ロジックツリーの課題ノード等） |
| タイトル文字 | `#111827` | メインタイトル |
| 本文文字 | `#1F2937` | 本文・太字 |
| 補足文字 | `#4B5563` | サブタイトル・補足 |
| ラベル文字 | `#6B7280` | 小さなラベル・推察ラベル |

**フォント:**
- `Noto Sans JP` を全スライドで統一使用
- サイズ目安：タイトル 27pt / 本文 15pt / ラベル 10pt / 大見出し 54pt

**レイアウトパターン:**
- **背景は`#FFFFFF`単色**（2層背景なし。白背景ベースでアクセントカラーを引き立てる）
- タイトルエリア下に`#42A4AF`のアクセントライン（高さ2pt）
- 右上に「株式会社エヌイチ」（15pt, `#4B5563`）
- カード型：`#F9FAFB`背景 + 左端に`#42A4AF`の細いバー
- フッターエリア：`#E5E7EB`の区切り線

**文字量（最重要）:**
- 各スライドのテキストは必要最小限
- キーワード＋短文（1行以内）
- 箇条書きは1項目1行、3〜5項目
- 図解・表を積極活用

**図解の活用:**
- **ロジックツリー**（課題スライドで使用。階層構造を左→右または上→下で表現）
- プロセス図 / フローチャート / ロードマップ
- 比較表（Before/After）
- カード型グリッドレイアウト
- ステップ図（番号付き矢印）

### 5-4. PPTX生成 → Google Drive アップロード手順

1. **python-pptx でPPTXを生成**
   - `output/proposal_{会社名}_{YYYYMMDD}.pptx` に出力
   - 上記デザインルールに厳密に従う
   - フォントは `Noto Sans JP` 固定

2. **Google Driveにアップロード（Google Slidesに自動変換）**
   - `mcp__google-workspace__uploadFileToDrive`（account: `main`）でアップロード
   - **`convertToGoogleFormat: true` を必ず指定する**（PPTXをGoogle Slidesに自動変換）
   - `fileName`: `{先方会社名}_ご提案書_{YYYYMMDD}`
   - **アップロード先フォルダID**: `18Ux5ehpsYQ9bjkFxstU46YwXmvJInRtq`（提案資料フォルダ）
   - ※ ローカルファイルのパスが `~/claude/output/` 配下の場合は `/Users/kyouyuu/Downloads/` にコピーしてからアップロードする（パス制限対策）

3. **Google SlidesのURLをユーザーに共有**
   - アップロード完了後に返ってくる `View Link` をそのまま共有する
   - Google Slidesとして直接編集・共有が可能

**ユーザーに確認:**
```
PPTXを生成し、Google Driveにアップロードしました。

- ファイル名: {先方会社名}様_ご提案書_{YYYYMMDD}.pptx
- スライド枚数: [N]枚（固定3枚 + 提案[M]枚 + 締め1枚）
- ローカル: output/proposal_{会社名}_{YYYYMMDD}.pptx
- Google Drive: [URL]

Google Driveで開いた後「Google スライドで開く」で編集できます。
確認して、修正点があればお知らせください。
```

→ 修正依頼があれば、python-pptxで該当スライドを更新し再アップロードする。
