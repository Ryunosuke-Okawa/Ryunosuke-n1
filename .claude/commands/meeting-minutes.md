# meeting-minutes — 商談議事録生成

商談の文字起こしテキストから、議事録生成・NAシート転記・Notion保存・フォローメール下書きを一括実行するスキル。

## トリガー

「**商談議事録**」のみ

## 入力ファイルの取得（起動時に必ず実行）

以下の優先順位で入力を取得する：

### ① ユーザーが直接提供した場合（スキップ）

以下のケースはファイル検出をスキップし、提供されたテキストをそのまま使用する：
- チャットにテキストを直接貼り付けた場合
- ファイルパスを指定した場合

### ② テキスト未提供の場合 → Google Drive Notta フォルダから自動検出

**MCPツール**: `mcp__google-workspace__listFolderContents`（account: `main`）
**フォルダID**: `1qS5ayYNas2Xw1v1VP24LdDqu-7X7Wjgy`

フォルダ内のファイル一覧を取得し、ユーザーに選択させる：

```
📄 Google Drive「Notta」フォルダから文字起こしファイルを検出しました：
1. 3_23 〇〇商談.txt（66 KB、15:47更新）
2. 3_23 △△打合せ.txt（83 KB、15:47更新）
どのファイルを使いますか？（番号で選択）
```

選択が確定したら、`mcp__google-workspace__downloadFromDrive`（account: `main`）でダウンロードし、Readツールで内容を読み込む。

**ファイル命名パターン（Notta）:**
- `M_D [名前].txt`（例: `3_19 マリンハウスモモ 高村さん 2回目.txt`）
- `会議名そのまま.txt`（例: `CINCA イベント(佐倉さん).txt`）
- `Zoom会議名.txt`（例: `3_3【エヌイチ(Kawaru）】お打合せリンク（西松建設株式会社 牧野様）.txt`）

ファイル名に `M_D` パターン（例: `3_23`）がある場合は会議日付として使用する。ない場合はファイルの更新日時またはファイル内容から日付を推測する。

## 実行手順

---

### STEP 1: 議事録生成

文字起こしテキストを解析し、商談議事録を作成する。

#### 前処理ルール
CLAUDE.mdに定義された以下のルールを必ず適用する：
- **ニックネーム→正式名変換**: 「ばつねじ」→住友駿仁、「日向」→小林日向 など
- **メールアドレス→氏名マッピング**: CLAUDE.mdのマッピング表を参照
- **社内用語補正**: 「リスティング数」→「リストイン数」、「パネル」→「ファネル」、「川村」→「河村」など
- **会社名の前株・後株**: 議事録中に登場する会社名は必ず正式名称（前株/後株を正確に判断）を使用する。不明な場合はWebSearchで確認する
- 外部参加者（クライアント）の名前が音声認識で不正確な場合は、文脈から推測して補正する
- 文字起こし特有の冗長表現（「えっと」「あの」等）は除去し、発言の意図を正確に要約する

#### 議事録フォーマット

```markdown
# [相手先名] 商談（第N回）議事録
日時: YYYY-MM-DD HH:MM〜HH:MM（推定可）
場所: オンライン（Zoom） / オフライン
参加者: [自社メンバー]、[先方メンバー（会社名・役職がわかれば記載）]

## アジェンダ
（会話の流れから主要トピックを抽出）

## 議論・決定事項

### 1. 経緯・背景
（なぜこの商談が発生したか、前回までの経緯）

### 2. 先方の課題
（先方が抱えている問題・困っていること）

### 3. 先方の希望
（先方がどうしたいか、何を求めているか）

### 4. サービス説明のポイント
（自社からどのような提案・説明をしたか）

### 5. 先方からの質問
（先方から出た質問とそれへの回答）

## ネクストアクション
| # | タスク | 担当者 | 期日 |
|---|--------|--------|------|

## 次回MTG
- 日時:
- 議題:

## 備考
（補足情報、注意事項など）
```

議事録は `output/` ディレクトリには保存せず、ユーザーに直接表示する。

### STEP 2: NAシート転記

議事録のネクストアクションをGoogle SpreadsheetのNAシートに転記する。

- **スプレッドシートID**: `1Ll5LsGL99n0P35msUqa23UgBvm9xgkE4Ke0Fn5Dxztw`
- **シート名**: `新規事業部`
- **MCPツール**: `mcp__google-workspace__appendSpreadsheetRows`を使用
- **フォーマット**: 既存行のフォーマットに合わせる
  - A列: タスク名（先方名を含めて具体的に記述）
  - B列: 期日（YYYY/MM/DD形式。会話から推測。不明なら1週間後）
  - C列: 担当者（大川龍之介 or 髙橋悠）
  - D列: 進行状況 → 常に「未着手」
  - E列: 備考（補足情報があれば）

転記前にユーザーに内容を確認し、承認を得てから実行する。

### STEP 3: Notion転記

商談議事録DBに該当ページを検索し、議事録を保存する。

**DB情報**:
- DB URL: `https://www.notion.so/1f8934e4905a80b3ab94fb3618bbd8ba`
- Data Source: `collection://1f8934e4-905a-8099-b301-000bcc647a8c`
- スキーマ: MTG（title）、日付（date）、ユーザー（person）、ステータス（status: 商談予定/商談済み/提案中/受注）

#### 3-1. 既存ページの検索（必ず指定DataSource内で確認）

**検索は必ず `data_source_url: collection://1f8934e4-905a-8099-b301-000bcc647a8c` を指定して行う。**

`notion-search` で企業名を検索する：

```json
{
  "query": "[企業名]",
  "data_source_url": "collection://1f8934e4-905a-8099-b301-000bcc647a8c",
  "filters": {}
}
```

ヒットしたページを `notion-fetch` で取得し、**企業名（MTGプロパティ）が一致するか**を確認する。

- 一致するページが見つかった → **3-2へ（既存ページに追記）**
- 該当企業のページが見つからない → **3-3へ（新規作成）**

#### 3-2. 既存ページがある場合

`notion-update-page` でページ末尾に今回の議事録を追記する。
（過去の商談履歴が蓄積されていく形）

#### 3-3. 既存ページがない場合（新規作成）

該当企業のページが存在しない場合のみ、`notion-create-pages` でデフォルトテンプレートから新規作成する：

- **親DB**: `https://www.notion.so/1f8934e4905a80b3ab94fb3618bbd8ba`
- **テンプレートID**: `1f8934e4-905a-80b7-9215-dca76030399d`（株式会社 テンプレート）
- **プロパティ（全項目を設定）**:
  - `MTG`: [企業名]（例: 「株式会社〇〇」）
  - `date:日付:start`: YYYY-MM-DD
  - `ステータス`: 「商談済み」
- **コンテンツ**: 議事録本文（Markdownフォーマット）を全文設定

#### 3-4. 完了報告

NotionページURLをユーザーに報告する。

### STEP 3.5: CRM転記

商談のNAをCRMスプレッドシートの該当企業行のNA列に追記する。

**スプレッドシート情報**:
- **スプレッドシートID**: `10X0S-zSXwm-oRqz3t7nKCFLvMyKdXGtEh_5PmBQipYo`
- **シート名**: `CRM`
- **企業名列**: B列
- **NA列**: O列

#### 手順

1. `mcp__google-workspace__readSpreadsheet`（account: `main`）でCRMシートを読み込み、B列で企業名を検索する
2. 企業名が部分一致する行を特定する（例: 「エヌイチ」→「株式会社エヌイチ」でもマッチ）
3. 該当行のO列（NA）の現在の内容を確認する
4. **追記ルール**:
   - 既存NAがある場合: 末尾に ` / YYYY/MM/DD [NAの内容]` を追記
   - 既存NAが空の場合: `YYYY/MM/DD [NAの内容]` を書き込む
5. `mcp__google-workspace__writeSpreadsheet` で更新する

**NA内容の書き方（議事録のネクストアクションから自社担当分を抜粋）**:
- 例: `2026/03/30 資料送付済み、次回4/10商談予定`

転記前にユーザーに確認を取る。

### STEP 4: フォローメール下書き作成

商談後のフォローアップメールをGmail下書きとして作成する。

#### 4-1. メールスレッド検索
- `mcp__google-workspace__searchGmail` で先方の会社名・担当者名を検索
- 該当スレッドが見つかったら `mcp__google-workspace__readGmailMessage` で最新メッセージを取得
- **取得する情報**: threadId, Message-ID, 先方メールアドレス, 件名

#### 4-2. メール本文作成

**MEMORY.mdのメールルールを厳守する。** 以下はすべてMEMORY.mdに定義済みのルールであり、毎回必ず参照・適用すること。

##### 冒頭フォーマット
- **宛先行**: `[会社名（正式名称）]\n[相手の名前] 様`
  - 既存スレッドの返信で会社名が自明な場合は省略可。その場合は「[名前] 様」から開始
- **挨拶**: `お世話になっております。\n株式会社エヌイチの大川です。`

##### 会社名の前株・後株ルール（重要）
- 必ず正式名称を使う。前株（株式会社〇〇）か後株（〇〇株式会社）かを正確に判断する
- 不明な場合はWebSearchで正式名称を確認してから使用する
- 議事録本文中の会社名にも同様に適用する
- 例: ファインディ → ファインディ株式会社（後株）

##### 本文構成
1. お打合せのお礼（第N回と明記）
2. 打合せ内容の振り返り（箇条書きで簡潔に）
3. ネクストアクション（先方・自社それぞれ）
4. 次回日程調整への言及
5. 締めの挨拶

##### 署名（MEMORY.mdから取得 — 変更しないこと）
```
―――――――――――――――――――――――――――――――
株式会社エヌイチ
Kawaru事業部
大川 龍之介（Okawa Ryunosuke）

TEL：080-2646-2420
Mail：r.okawa@n1-inc.co.jp
Website：https://n1-inc.co.jp/

「AI（アイ）ある社会に 」
―――――――――――――――――――――――――――――――
```

##### スレッド引用（返信メールの場合）
- 本文・署名の下に元メールを引用形式（`>`）で含める
- 形式: `YYYY年M月D日(曜) 送信者名 <email>:\n\n> 本文...`

##### Cc
- `y.takahashi@n1-inc.co.jp`（髙橋悠）を常に含める

#### 4-3. Gmail APIで下書き作成

google-workspace MCPツールは日本語件名を文字化けさせるため、**PythonでGmail APIを直接呼び出す**。

```python
# 実行するPythonスクリプトの概要
import json, base64
from email.mime.text import MIMEText
from email.header import Header
from google.oauth2.credentials import Credentials
from googleapiclient.discovery import build

# 認証: ~/.google-mcp/tokens/main.json から読み込み
# MIMEText で本文組み立て
# Header(subject, 'utf-8') で件名エンコード
# In-Reply-To, References に元メッセージのMessage-IDを設定
# threadId を指定してスレッド返信に
# base64.urlsafe_b64encode → users().drafts().create()
```

下書き作成後、以下を報告する：
- 宛先・Cc
- 件名
- スレッド紐づけ状態
- 「Gmailの下書きフォルダからご確認・送信をお願いします」

## エラーハンドリング

- **NAシートのMCPツールが使えない場合**: NAの内容をテキストで表示し、手動転記を依頼
- **Gmailスレッドが見つからない場合**: 先方のメールアドレスをユーザーに確認。新規メールとして下書き作成
- **Python実行エラー**: エラー内容を表示し、google-workspace MCPツールでの代替（件名文字化けの注意付き）を提案
