# 開発MTG 議事録

日時: 2026-05-12（約57分）
場所: オンライン（Zoom）
参加者: 大川龍之介（Kawaru事業部）、高木直人（戦略設計／プラン設計）、新田雄輝（エンジニア／インフラ・決済）、木村圭佑（エンジニア／連携・プロンプト）
議事録担当: 大川龍之介

## アジェンダ
1. 自己紹介（新田・高木の初顔合わせ）
2. 各メンバーの現状タスク把握（全体俯瞰）
3. 5/26 β版ローンチに向けた優先順位と取捨選択
4. 料金プラン設計（フリープラン vs 2週間無料トライアル）
5. Function Calling 導入検討
6. クレジット制プラン案の検討
7. 共有・相談事項

## 議論・決定事項

### 1. 自己紹介
- 高木直人：U3の代わりに上層・戦略設計のレイヤーに入る。価格・プラン設計、プロンプト周りの整理を担当。AI関連プロダクト開発と教育系新規事業の経験あり。
- 新田雄輝（Nitt）：4月1週目から参画。元フリーランスエンジニア、PHPメインのWeb開発が中心。直近はAIワークフロー系・HRテックの経験豊富。木村経由でジョイン。

### 2. 新田の担当領域：インフラ整備＋PayPal決済実装
- 背景: 従来「デモ・社内・本番」が単一環境で本番運用に耐えない状態だった。
- 議論（新田）:
  - 4月後半までで検証環境／本番環境を分離し、IaCで冗長化・堅牢化を実施。ユーザー急増にもパラメータ調整で対応可能な構成にレビュー中。
  - 5月入ってからは PayPal によるサブスク決済画面を実装。アカウント画面に「課金管理」ボタンを追加し、プラン選択→PayPal契約まで動作する状態を昨日完成。レビュー段階。
  - 構成図・コスト感の資料は後日共有予定。
- 決定:
  - **インフラ移行のタイミングは5/26リリースとの兼ね合いで再調整**（U3さんと相談）。
  - **PayPal決済の細部（解約／トライアル18日経過後の自動課金／課金失敗時のアラート通知など）は引き続き新田が実装**。

### 3. 木村の担当領域：ツール連携＋プロンプト品質
- 背景: 連携対象ツールの申請・審査待ちが複数進行中。
- 議論（木村）:
  - 連携は基本的に完了。今は「会話から思った通りのワークフローブロックが正しく出てくるか」のクオリティ調整フェーズ。
  - メール・インフラ周りのアラート系はまだ未着手で、Backlog で整理予定。
- 決定:
  - 課金周りに合わせて「ユーザー登録時に追加で取得すべき情報」を後日設計（マーケ視点で必要な項目を整理）。

### 4. 料金プラン設計（最優先課題）
- 背景: 現状は「2週間無料トライアル→自動有料課金」フェーズ。高木より「半永久に使えるフリープランを設けるか」提案あり。
- 議論:
  - 新田: 「画面構成（クレカ登録のマスト／非マスト）に直結するため、フリープランの有無は早急に決めてほしい。プラン額・期間の増減は後からでも容易」。
  - 木村: 同意。フリープランの有無＋プランごとの機能制限（例：ワークフロー数の上限）が決まれば実装は容易。
  - 高木: ChatGPT 的に常に無料で触れる枠を想定。ただし直前変更は技術コストが高いため早期決定が必要。
- **決定**:
  - **高木が「フリープランあり／なし」を早急にフィックスして提案。プランごとの機能制限（ワークフロー数等）もセットで設計する**。

### 5. プロンプト改修と Function Calling 導入検討
- 背景: 高木が壁打ちで作成したプロンプト改修案を共有。「目次承認フロー」「Function Calling 化」の2提案。
- 議論:
  - **目次承認フロー**: 木村「Kawaru は『迷わせず一気に作る』思想なので、提案→承認ではなく即作成方式にしている」。高木も納得し、当該提案は撤回。
  - **Function Calling 化**: 当初は会話が噛み合わず認識すり合わせ。新田の整理で「Gemini SDK 側でツール定義を渡し、AI が必要に応じて外部ツールを呼び出す MCP 的なアプローチ」と共通理解を形成。
  - メリット: 連携ツール／SaaS が増えてもプロンプトを無限に膨らませる必要がなくなる。エージェント的に自律的なワークフロー生成へ拡張しやすい。
- **決定**:
  - **木村が Function Calling 方式を試作・検証**。Python 関数方式／JSON 方式どちらでも実装可能か含めて確認。
  - 既存プロンプトの編集権限を木村が高木に付与し、**高木がプロンプト本体の修正を進める**。
  - **5/26 までに Function Calling が間に合えば移行、間に合わなければ既存方式のままリリース**。

### 6. 5/26 ローンチに必要な機能の整理
- 背景: 5/26 はエヌイチ SNS（スクール事業部アセット活用）で告知。目標 500 名、個人ユーザー中心、組織導入の取りこぼしを後追いで拾う方針。
- **決定**:
  - 5/26 時点で必要なのは「個人クレカ課金が成立する状態」と「プロンプト出力の質」。
  - **請求書払い・インボイス対応は 6 月 2 週目目処**（展示会＝6/3–4 のリード経由で法人提供が始まるため）。
  - 管理画面（エヌイチ側）の整備は β 版期間中はスコープ外。

### 7. クレジット制プラン案（高木提案）
- 背景: LLM ヘビーユース時の利益率圧縮を抑止するため、ワークフロー単位でクレジットを消費する方式を構想。
- 議論:
  - 高木案: 1 ライセンス＝1,000 クレジット／月。メール下書き＝1 クレジット、請求書作成等の重ワークフロー＝3 クレジット等。超過時は追加購入可能。
  - 新田: PayPal 側の追加課金機能としては実装可能。ただし「事前にクレジット数を確定できるか」が論点（入力本文の長さで実コストが変動するため）。動的決定 vs 決め打ちで設計が大きく変わる。
  - 大川: クレジット制は「定期実行ワークフローが暴走的に回り続ける」リスクの抑止にもなる。BiS／コンサル側でサポートする論点でもある。
- **決定**:
  - **高木が壁打ちのうえ、「決め打ちでこのくらい」という具体案を再提案**。OKであれば実装フェーズへ。

### 8. 会員登録時の取得情報
- 議論:
  - 現状は Google／Microsoft ログインで「名前＋メアド」のみ取得。
  - 木村: マーケ的に追加で取りたい情報があれば登録時に取れる。
  - 大川: 料金体系の決定に依存。有料化（請求書発行）時に会社情報を取得する案で十分かも。
- **決定**:
  - **料金プラン決定後に追加項目を確定**。

### 9. ワークフローのインポート／エクスポート機能
- 議論（大川→木村）:
  - エクスポートは YAML 形式で既に実装済み。
  - **インポート機能はまだ未実装だが、実装は容易**。
