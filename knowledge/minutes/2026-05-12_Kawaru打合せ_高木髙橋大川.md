## 第1部：料金プラン・プライシング議論（髙橋さん同席）

### 1. 料金プラン全体像（高木さんからの説明）

- **プラン構成**
  - フリー：30クレジット/月（30回ワークフロー実行）
  - パーソナル：1,000クレジットで4,980円
  - チーム（主力）：5ライセンス以上、1ライセンス9,800円、初期費用 最低20万円
  - エンタープライズ：個別交渉
- **初期費用2パターン**
  - 20万円（ライト）：ハンズオン形式で、担当者自身が使いこなせるようになる
  - 30万円：業務をヒアリングしAs-Isを作成、よりインパクトの高い業務にハンズオン5本
- **月次オプション**
  - 月次コンサル：10万円（いつでも質問OK）
  - 部署展開：30万円
  - プレミアムコンサル：50万円（業務そのものの再設計＋定着支援）
- **クレジット**：軽い処理1、重い処理3〜5。1人1,000クレジット/月

### 2. 売上想定（展示会経由8社契約モデル）

- 月次粗利：約31.8万円（ライセンス収入のみ）
- インフラ固定費：約3.75万円
- 無料ユーザー控除分：約28万円
- 6ヶ月売上合計目標：1,200万円／粗利860万円
- 1社あたり6ヶ月後のライセンス数：20ライセンス程度を想定

### 3. 髙橋さんからの主要論点

#### 論点①：月次コンサルが選ばれない／解約リスク

- 「業務特定の難しさ」は顧客には伝わりにくく、「自分でできる」と思って月次コンサルを頼まない会社が多いのでは
- 結果、Kawaruが「使えない」と判断されて解約に至るリスクがある
- → **月次コンサルの一部をライセンス費に内包する／最初の3ヶ月で集中支援してやめられない状態を作る**という代替案を提示

高木さんの返答：
- 同じ議論を内部で検討済み。ライセンス費を1万2,800〜1万4,000円に上げる試算もしたが、「コンサル不要になった後にライセンス費が高いままなのが嫌」という判断で現案に
- ライセンス数による稼働時間差（5ライセンスと20ライセンスでコンサル工数は変わらない）を考慮し、別建てを選択
- → **最終的には「決めの問題」と認識**

#### 論点②：見せ方を「スターターセット化」する（大川提案）

- 月次コンサル3ヶ月分をスターターセットとして組み込み、「本来13万のところ9.8万でやります」という割引提示
- → **高木さん：「最初に決断させる根拠としてディスカウントを使うのは非常にいい視点」と評価**
- → 中堅・大企業は予算組みの都合上、最初に半年分のパッケージを提示する戦略が必要
- → スタートアップ／ベンチャーにはシンプルなライセンス費＋初期費用の見せ方を維持

→ **顧客セグメントによって提示する料金構造を分ける方針で合意**

#### 論点③：ハンズオン「5本」という記載のリスク

- ワークフローには重い／軽いの差が大きく、「5本」を約束するのはリスクが高い
- → 高木さん：マトリックスでヒアリング→「効率化インパクトが高い×実装が簡単」なものから着手するロジックをエージェント化済み。誰がやっても標準化可能
- 工数換算は5,000円/h固定で試算しているが、エージェント化部分が多いため利益率はさらに上がる見込み
- → 表記の「5本」は要再検討（具体本数のリスクを避ける書き方へ）

#### 論点④：クレジット設計を顧客にどこまで見せるか

- 高木さん：基本的にユーザーには見せない。25人で2万5,000クレジットあれば通常業務はカバー可能。足りなくなったら追加できる、という設計

#### 論点⑤：トップライン2〜3億円の実現可能性

- 髙橋さん：展示会8社×6ヶ月で1,200万→年商換算でも限界2億程度。3億は遠い
- 高木さんはKawaruを「フロント商材」と位置付け、ここ単体で大規模売上は想定していない
- → **共通認識：Kawaruは"フック商材"として活用し、本マーケットは複数商材で2〜3億を作る前提**
- → ただし、フック商材としても2〜3億のハードルは現状では厳しい認識を共有

#### 論点⑥：競合（JAPAN AI／神馬）との比較

- 髙橋さんの懸念：JAPAN AIが伸び、神馬が伸びていない理由がまだ言語化できていない
- 髙橋さん仮説：JAPAN AIは8個目のプロダクトとしてローンチ→既存大型クライアントへのアップセル戦略が効いている可能性
- → **高木さんが調査を担当。理由を分析し、Kawaruの設計に反映する。明確化できなければマーケットニーズに沿った独自の企業努力で勝負**

### 4. 決定事項（第1部）

| # | 内容 |
|---|------|
| 1 | 料金プランの基本構造（フリー／パーソナル／チーム／エンタープライズ）は現状維持 |
| 2 | 顧客セグメントによって料金提示パターンを分ける（中堅大企業＝半年パッケージ、SMB＝シンプル提示） |
| 3 | 「スターターセット」として3ヶ月コンサル込みディスカウントを採用検討 |
| 4 | ハンズオン「5本」表記は再検討（具体本数を約束しない見せ方へ） |
| 5 | JAPAN AI／神馬の伸び要因を高木さんが調査 |

## 第2部：開発進捗共有（髙橋退席後）

> **以下は髙橋さんへ別途共有する内容として整理。要点を簡潔にまとめています。**

### 5/8 木村さんMTG以降の進捗（高木さんから）

#### ① LLMの動的ルーティング設定を実装完了

- 課題：Kawaruは全てGemini Flashで動作していて、複雑な処理（ワークフロー生成等）で精度不足
- 対応：用途別にGemini Pro／Flashを動的に切り替えるルーティング設定を高木さんが設計→木村さんが実装
- 結果：ワークフロー作成時はPro、簡易サマリーはFlash等の使い分けが可能に。**コストを抑えつつ精度を向上**

#### ② プロンプト一括書き換えを実施

- 木村さんから「ブロックタイプ一覧」を共有してもらい、Kawaru内部のプロンプト（メール返信、返信判定、ワークフロービルダー等）を高木さん側で全面リライト済み
- 現在は木村さんに反映を依頼中（プロンプト書き換え権限が高木さんにないため）

#### ③ ユニファイドAPI導入は見送り

- SaaS連携の保守運用のために検討していたユニファイドAPIは、国産SaaSメインの利用を踏まえると**インパクト小→導入見送り決定**
- 木村さんは引き続き個別SaaSの繋ぎ込みを継続

### 今後着手したい3つの構造改善（高木さんから提案）

現状のプロンプト改修で精度は1.5〜2倍。さらに飛躍させるには以下3点が必要：

| # | 施策 | 効果 |
|---|------|------|
| 1 | **プランニング2段階化（Plan-then-Execute）** | 「目次を出して承認→詳細実装」の2段構成にすることで、意図と違う成果物が出来上がる事故を防止 |
| 2 | **Function Calling対応** | Geminiが扱いやすいJSON構造に書き換え。エラー激減＝触り心地が大幅向上 |
| 3 | **Few-shotの動的注入（RAG）** | 例示データ（メール例100件等）を動的注入してアウトプット精度を底上げ |

→ Function Callingまでβ版で着手したい意向

### 大川さんからの確認事項

#### 確認①：トリガー条件・分岐条件の少なさは？

- 大川「メールの下書きを朝8時に実行、等のトリガー設定ができない」
- 高木「これはもっと手前の話。プロンプトの構造的な土台が整っていないと話にならない。土台が整えば、選択肢追加もユーザー指定の柔軟対応も簡単」
- → **プロンプト構造改善が先。トリガー追加は後フェーズ**

#### 確認②：木村さんとの優先順位すり合わせができていないのでは？

- 現状：高木さんが自分の判断で「重要×緊急」のタスクを木村さんに当てている。**木村さん側のタスク優先度には介入していない**
- 大川「木村さん／新田さん側の優先順位を共有してもらった上で、すり合わせした方が手戻りリスクが下がる」
- 高木「全く同感。本日14時の開発MTGでクリアにしましょう」
