## 2. 実績ハイライト（KPIバー）

DigiRise は5指標で殴っているが、当社は数字が弱い。3つの方針案：

### 方針① 数字で殴れる項目に絞って小さく出す
- Kawaru シリーズ受講者数（要数字確認）
- 自社業務での自動化率：**9 割超**（代表が実証）
- 自動化対象業務数：**12 業務以上**（活用事例数と整合）

### 方針② 数字を全部捨てて「定性メッセージ」に置換
> "数で語らない。一社にどこまで踏み込めるかで選ばれている。"

ヒーロー下のバーで「Kawaru事業／代表自ら実装／完全カスタム」の3点だけ見せて、KPIブロック自体を**廃止**。

### 方針③ 「自社事例」を数字化して並べる
代表・大川 / エヌイチ自社が Claude Code で達成した数値を、堂々と一次ソースとして出す。

| 指標 | 数値 | 補足 |
|---|---|---|
| 自動化業務数 | 12 業務超 | 議事録・メール・提案書・経理 ほか |
| 削減工数 | 月平均 ◯◯ 時間 | 自社実測（要再計算） |
| 自動化率 | 90% | 代表業務ベース |

> 👉 大川さん判断ポイント：**方針①②③のどれで行くか**。私のおすすめは **方針③（自社事例を一次ソースで数値化）**。「他社実績は薄いが、自社で誰よりも徹底的に使い倒している」というポジショニングが取れる。

## 3. こんな課題ありませんか（課題提起）

> 「Claude Code を導入したい」ではなく、その先で詰まる課題に当てる。

```
01  社内に Claude Code を入れたいが、何から始めるべきかわからない
02  情報収集はしたが、自社の業務にどう落とすかが見えない
03  一部のエンジニアだけが使い、組織全体の生産性に繋がっていない
04  Codex との使い分け・連携が整理できておらず、効果が分散している
```

---

## 4. なぜいま Claude Code × Codex か

> 1コラム文章でストーリーテリング。ここで Codex 連携の必要性を立てる。

たたき：

> 2026 年、開発生産性の主役は「単一の生成 AI」から「複数 AI の組み合わせ」へ移った。
> Claude Code は文脈理解と長期タスク遂行力で群を抜き、Codex はコード生成のスピードと言語横断性で頭ひとつ抜けた。
> 両者を**役割分担して動かせる人材**を持つ組織だけが、来年以降の開発生産性で勝ち残る。
>
> 本研修は、その「組み合わせて動かす」現場運用ノウハウを、貴社の社員に直接インストールするためのプログラムです。

## 5. 研修プログラム概要

> DigiRise は 2 本柱（研修＋コンサル）で見せているが、**このLPは研修1本に絞る**。コンサル相当は BPO LP で別建て。

```
TRAINING — Claude Code / Codex 研修プログラム

社員に Claude Code と Codex を "自分の業務道具" にしてもらう、完全カスタム型の法人研修。
講義より「翌日使える状態」を優先し、貴社のコードベース・業務題材で実機ハンズオン。

✓ 基礎〜実機操作までを段階設計
✓ 非エンジニア／エンジニア別クラスに対応
✓ Codex 連携・MCP・Hooks まで網羅
✓ 集合 / オンライン / ハイブリッド対応
```

## 6. 3ステップカリキュラム（看板）

Kawaru Team の 3 ステップを Claude Code 文脈に転用。

```
STEP 01  Claude Code / Codex を "自分ごと" で理解する
業務題材に置き換えた講義設計。「ツールの使い方」ではなく
「あなたの仕事がどう変わるか」にフォーカス。

STEP 02  実機ハンズオンで動かす
貴社のコードベース・業務題材を使った半日 / 1 日ハンズオン。
Claude Code + Codex の使い分けを体で覚える。

STEP 03  チームで成果物を出すワークショップ
実務課題をテーマに、参加者が AI で成果物を作る。
研修終了時点で "使える成果物" が現場に残る完全実践型。
```

## 13. FAQ

